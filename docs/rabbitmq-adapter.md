# RabbitMQ Adapter Documentation

Module: `spring-boot-cqrs-rabbitmq`  
Package: `com.borjaglez.cqrs.rabbitmq`

The RabbitMQ module provides distributed implementations of all three buses. When added to the classpath, it auto-configures RabbitMQ exchanges, queues, bindings, consumers, and publisher beans.

## Architecture

```
Producer Service                          Consumer Service
+-------------------+                    +-------------------+
| CommandBus        |                    | CommandConsumer    |
|  (RabbitMq)       |----> Exchange ---->| Queue             |
|                   |      (topic)       |  -> Registry      |
| EventBus          |                    | EventConsumer     |
|  (RabbitMq)       |----> Exchange ---->| Queue             |
|                   |      (topic)       |  -> Registry      |
| QueryBus          |                    | QueryConsumer     |
|  (RabbitMq)       |----> Exchange ---->| Queue             |
|                   |      (topic)       |  -> Registry      |
+-------------------+                    +-------------------+
```

Each bus type has its own exchange. Messages are routed using topic exchange routing keys derived from the message name (via `MessageNamingStrategy`).

## Exchanges, Queues, and Routing

For commands and events, the module creates three exchange/queue sets:

| Purpose | Exchange | Queue |
|---|---|---|
| Main | `{prefix}.{exchangeName}` | `{prefix}.{appName}.{exchangeName}` |
| Retry | `{prefix}.{exchangeName}.retry` | `{prefix}.{appName}.{exchangeName}.retry` |
| Dead Letter | `{prefix}.{exchangeName}.dead_letter` | `{prefix}.{appName}.{exchangeName}.dead_letter` |

For queries, only the main exchange and queue are created (no retry or dead-letter).

All exchanges are **topic exchanges**. Routing keys are the message names generated by `MessageNamingStrategy`.

Example with `prefix=order-service`, `appName=my-app`, `exchangeName=commands`:

```
Exchange:    order-service.commands
Queue:       order-service.my-app.commands
Retry Ex:    order-service.commands.retry
Retry Queue: order-service.my-app.commands.retry
DLX:         order-service.commands.dead_letter
DL Queue:    order-service.my-app.commands.dead_letter
```

Bindings are created for each registered handler's routing key. For example, if a service has handlers for `CreateOrderCommand` and `ConfirmOrderCommand`, two bindings are created on the main queue.

## Bus Implementations

### RabbitMqCommandBus

Implements `CommandBus`. Publishes commands to the command exchange.

- `dispatch(command)` -- publishes fire-and-forget with message type `"command"`
- `dispatchAndReceive(command)` -- uses `RabbitTemplate.sendAndReceive()` for RPC-style request/reply with message type `"command_reply"`

### RabbitMqEventBus

Implements `EventBus`. Publishes events to the event exchange.

- Has a **fallback** to the local `SpringEventBus` if the RabbitMQ connection fails (catches `AmqpException`)
- This ensures events are still processed locally when RabbitMQ is unavailable

```java
public void publish(Event event) {
    try {
        publisher.publish(exchange, routingKey, event, "event");
    } catch (AmqpException e) {
        log.warn("Failed to publish via RabbitMQ, falling back to local bus");
        fallbackEventBus.publish(event);
    }
}
```

### RabbitMqQueryBus

Implements `QueryBus`. Uses `sendAndReceive()` for synchronous request/reply with message type `"query"`.

### RabbitMqPublisher

Shared publisher component that wraps `RabbitTemplate`:

- `publish(exchange, routingKey, message, type)` -- one-way publish with a `cqrs.message.type` header
- `publishAndReceive(exchange, routingKey, message, type)` -- RPC-style send/receive; checks for a `cqrs.error` header on the reply

## Retry and Dead-Letter Queue Strategy

Commands and events use a three-tier queue strategy:

```
1. Message arrives at Main Queue
2. Consumer attempts to process it
3. On failure:
   a. If redelivery count < maxRetries:
      - Increment cqrs.redelivery.count header
      - Send to Retry Exchange (with # routing key)
      - Retry Queue has TTL, after which message returns to Main Exchange (via DLX)
   b. If redelivery count >= maxRetries:
      - Send to Dead Letter Exchange
      - Message stays in Dead Letter Queue for manual inspection
```

Configuration:

```yaml
cqrs:
  rabbitmq:
    retry:
      max-attempts: 3    # default
      ttl: 1000           # milliseconds, default
```

The retry queue is configured with:
- A TTL (time-to-live) that causes messages to expire after the configured delay
- A dead-letter exchange pointing back to the main exchange, so expired messages are re-delivered

This creates an exponential-backoff-like pattern where failed messages are retried after a delay.

## Consumer Configuration

Each bus type has its own `SimpleMessageListenerContainer` with configurable concurrency:

```yaml
cqrs:
  rabbitmq:
    commands:
      concurrent-consumers: 10    # min threads
      max-concurrent-consumers: 20  # max threads
    events:
      concurrent-consumers: 10
      max-concurrent-consumers: 20
    queries:
      concurrent-consumers: 10
      max-concurrent-consumers: 20
```

### Consumer Classes

| Consumer | Handles | Behavior |
|---|---|---|
| `RabbitMqCommandConsumer` | Commands | Routes to `CommandHandlerRegistry`; supports fire-and-forget, reply, and wait modes |
| `RabbitMqEventConsumer` | Events | Routes to `EventHandlerRegistry`; fire-and-forget only |
| `RabbitMqQueryConsumer` | Queries | Routes to `QueryHandlerRegistry`; always returns a reply |

All consumers extend `RabbitMqConsumer`, which provides the `handleConsumptionError()` method for retry/dead-letter logic.

Message types (set via the `cqrs.message.type` header):

| Type | Behavior |
|---|---|
| `command` | Fire-and-forget; errors trigger retry/dead-letter |
| `command_reply` | RPC; result is returned to caller; errors are propagated |
| `command_wait` | Synchronous; returns `"OK"` on success; errors are propagated |
| `event` | Fire-and-forget; errors trigger retry/dead-letter |
| `query` | RPC; result is returned to caller |

## Naming Strategy

### RabbitMqNamingStrategy

```java
public interface RabbitMqNamingStrategy {
    String exchange(String exchangeName);
    String exchangeRetry(String exchangeName);
    String exchangeDeadLetter(String exchangeName);
    String queue(String applicationName, String exchangeName);
    String queueRetry(String applicationName, String exchangeName);
    String queueDeadLetter(String applicationName, String exchangeName);
}
```

### DefaultRabbitMqNamingStrategy

Prefixes all names with the configured prefix:

| Method | Pattern | Example (`prefix=cqrs`, `appName=my-app`, `exchange=commands`) |
|---|---|---|
| `exchange()` | `{prefix}.{exchange}` | `cqrs.commands` |
| `exchangeRetry()` | `{prefix}.{exchange}.retry` | `cqrs.commands.retry` |
| `exchangeDeadLetter()` | `{prefix}.{exchange}.dead_letter` | `cqrs.commands.dead_letter` |
| `queue()` | `{prefix}.{app}.{exchange}` | `cqrs.my-app.commands` |
| `queueRetry()` | `{prefix}.{app}.{exchange}.retry` | `cqrs.my-app.commands.retry` |
| `queueDeadLetter()` | `{prefix}.{app}.{exchange}.dead_letter` | `cqrs.my-app.commands.dead_letter` |

You can provide a custom `RabbitMqNamingStrategy` bean to override the default naming.

## Auto-Configuration

The module provides four auto-configuration classes:

| Class | Condition | Creates |
|---|---|---|
| `RabbitMqCqrsAutoConfiguration` | `RabbitTemplate` on classpath + `cqrs.rabbitmq.enabled=true` | `RabbitMqNamingStrategy`, `RabbitMqPublisher`, `RabbitMqBusDeclarationBuilder` |
| `RabbitMqCommandBusAutoConfiguration` | Above + `CommandHandlerRegistry` bean | `RabbitMqCommandBus`, command `Declarables`, command listener container |
| `RabbitMqEventBusAutoConfiguration` | Above + `EventHandlerRegistry` bean | `RabbitMqEventBus`, event `Declarables`, event listener container |
| `RabbitMqQueryBusAutoConfiguration` | Above + `QueryHandlerRegistry` bean | `RabbitMqQueryBus`, query `Declarables`, query listener container |

The bus auto-configurations use `@Value("${spring.application.name:cqrs-app}")` for the application name used in queue naming.

## Docker Setup

The `example-rabbitmq` application uses [Spring Boot Docker Compose](https://docs.spring.io/spring-boot/reference/features/docker-compose.html) to automatically start and stop RabbitMQ when running the application. No manual `docker compose up` is needed.

Minimal `compose.yml` for local development:

```yaml
services:
  rabbitmq:
    image: rabbitmq:3-management
    ports:
      - "5672"
      - "15672"
    environment:
      RABBITMQ_DEFAULT_USER: guest
      RABBITMQ_DEFAULT_PASS: guest
```

Spring Boot configuration:

```yaml
spring:
  docker:
    compose:
      file: examples/example-rabbitmq/compose.yml
      lifecycle-management: start-and-stop
```

Spring Boot Docker Compose auto-configures `spring.rabbitmq.*` properties (host, port, username, password) from the running container — no manual connection configuration is required.

See the [example-rabbitmq](../examples/example-rabbitmq) application for a complete working setup.
